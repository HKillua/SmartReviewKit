# Phase 3：Artifact Post Actions

## 背景问题

如果 Agent 只是“回答给你听”，学习闭环并不完整。  
尤其是在复习、章节深入讲解和错题复盘场景里，用户往往还需要：

- 笔记
- 闪卡
- 复习计划

如果这些成果物完全靠用户自己整理，Agent 的学习辅助价值会明显打折。

## 为什么要这样改

这一步本来有两种方向：

- 直接接外部 Notion / Anki / 日历
- 先把成果物做成前端可下载产物

第一版选择了后者，因为这样更稳、更容易演示，也不会引入外部账号和平台依赖。  
同时在结构上仍然保留了“MCP-shaped post-action”思路，为后续真实外部集成留接口。

## 核心思路

核心思路是把“后动作”从主回答链路里拆出来。

主回答完成后，系统根据 SkillPolicy 的 `post_actions` 决定是否生成：

- `notes_export`
- `flashcard_export`
- `schedule_export`

这些产物不会改变主任务输出，只是在主任务成功后附加生成，并把下载信息回传给前端。

## 运行时链路怎么变化了

升级后，链路变成：

主任务回答完成 → 后台触发 post action adapter → 产物写入对象存储 / 本地存储 → `done` metadata 带 `artifacts` → 前端通过下载接口获取文件

这里最关键的是：

- 主回答和后动作解耦
- 后动作失败不会让主回答失败
- 用户可以把学习结果真正带走

## 关键 tradeoff

这一步最大的取舍是：**先做前端可下载产物，不先做真实第三方同步**。

好处：

- 实现路径短
- 故障面小
- 非常适合演示和面试讲解
- 后面要接真实 MCP 时，adapter 形态已经在了

代价：

- 现在还不是完整外部生态闭环
- 只能说是“面向真实 MCP 的过渡版本”

但这是一个合理的第一版，因为它先把“成果物”这件事真正做成了可交付能力。

## 效果与验证

这一步的直接效果是：

- `done` metadata 里会带 `artifacts`
- 前端可以通过下载接口拿到笔记、闪卡或复习计划
- 后动作失败时，主回答仍然成功返回

这说明系统已经不只是“会回答”，还会“顺手产出学习成果物”。

## 面试时怎么讲

可以直接这样讲：

> 我没有第一版就把系统强耦合到 Notion、Anki 或日历，而是先把后动作抽象成 MCP-shaped adapter，让 Agent 在主任务结束后生成可下载的学习成果物，比如笔记、闪卡和复习计划。这样主回答链路保持稳定，后动作失败也不会拖垮主任务，同时又为后续真实外部 MCP 集成留出了清晰接口。
