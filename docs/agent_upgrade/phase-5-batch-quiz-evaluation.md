# Phase 5 - Batch Quiz Evaluation

## 背景问题

前面的学习 Agent 已经支持出题、判题、错题记录和节奏控制，但判题链路一直默认“单题心智模型”：

- 用户一次只提交一道题
- 工具只返回一份 verdict / score / explanation
- 记忆和 pacing 也按“一次判题 = 一个结果”理解

这会在真实学习场景里暴露出两个明显问题：

- 学生经常会一次性回答两题甚至多题，系统只能粗粒度整体评估
- 即使上一轮刚出完题，下一轮学生只发“第一题我选 A，第二题我选 B”，系统也不一定能稳定对齐

如果不解决这件事，后面的错题本、知识掌握度更新和学习节奏判断都会被一个总评误导。

## 为什么要这样改

这次升级的核心目标，不是让系统“能勉强处理多题”，而是把判题正式升级成一条批量学习闭环：

- 一次工具调用支持多题逐题评分
- 支持接上一轮 quiz_generator 的题目上下文继续作答
- 支持自由文本连续作答时的结构化拆题
- 逐题写回错题与知识图谱
- pacing 不再只看一个总 verdict，而是按题级结果判断

换句话说，这次不是单点功能增强，而是把“判题”从单题工具升级成真正可服务学习场景的批量能力。

## 核心思路

这次采用的是“**规则优先，LLM 兜底；对齐成功才评分，否则先澄清**”的策略。

整体分成三层：

1. **输入对齐层**
   - 先解析显式编号结构
   - 再尝试对齐最近一轮 quiz_generator 题目
   - 最后才用 LLM 做自由文本结构化拆题

2. **批量评分层**
   - `quiz_evaluator` 统一走 `items` 批量 schema
   - 内部仍复用单题判分、证据检索和解释增强能力
   - 对外返回一次批量工具调用结果

3. **学习闭环层**
   - 错题本按题回写
   - `knowledge_map` 按题、按 concept 更新
   - pacing 优先读结构化 `batch_results`

这样设计的目的，是把自由文本理解和最终评分分开：  
LLM 可以参与拆题和对齐，但不直接跳过结构化阶段去“整段脑补评分”。

## 运行时链路怎么变化了

升级后的判题链路变成：

1. 用户发来批改请求
2. Agent 仍然把它视为 `quiz_evaluator`，不新增新的 task intent
3. 在真正调用工具前，先进入统一的批量题答对齐器
4. 对齐器依次尝试：
   - 显式编号解析
   - 最近一轮出题上下文对齐
   - 自由文本 LLM 结构化拆题
5. 如果题目和答案能稳定一一对应，则进入批量评分
6. 工具返回：
   - 总评
   - 平均分
   - 每题 verdict / score / explanation / concepts
   - `batch_results` 等结构化 metadata
7. 后续记忆和 pacing 直接消费逐题结果

对用户来说，最终感知到的是：

- “一次交两题/三题，系统会逐题批改”
- “如果系统无法稳定识别题答对应关系，不会乱判，而是明确要求补充题号或原题”

## 关键 tradeoff

这次最核心的 tradeoff 有四个。

第一，**不改 Planner 语义**。  
多题判改仍然属于 `quiz_evaluator`，不把“多题”额外建成一个新 intent。这样保持了主链路稳定，复杂度集中在工具入口。

第二，**规则先行，LLM 后置**。  
显式编号和最近一轮出题上下文能解决的情况，优先不用 LLM。只有当用户真的以自由文本连续作答时，才调用 LLM 做结构化拆题。

第三，**宁可澄清，不做部分静默判题**。  
如果当前消息和最近题目上下文不足以稳定一一对齐，就返回澄清，而不是偷偷只判识别出来的那一题。这是为了保证学习反馈的可信度。

第四，**v1 控制上限为 5 题**。  
批量能力要有边界，避免一次性塞进过多题目把延迟和对齐难度都拉高。超过 5 题时明确要求拆分。

## 效果与验证

这次升级完成后，系统具备了三类新增能力：

- 显式双题/三题批量判改
- 接上一轮 quiz_generator 题目继续作答并自动对齐
- 自由文本连续作答时的 LLM 结构化拆题

验证分了三层：

- 单测与 runtime 回归
  - `tests/unit/test_phase5_batch_quiz.py`
  - `tests/unit/test_quiz_evaluator_grounding.py`
  - `tests/unit/test_phase4_agent_features.py`
  - `tests/unit/test_agent_trace_runtime.py`
  - `tests/unit/test_phase_s.py`
  - `tests/unit/test_eval_runner.py`

- 关键结果
  - 相关单测与回归共 `88 passed`
  - `python -m tests.eval.runner` 跑完 `16/16` 全通过

- 行为评测新增场景
  - 显式双题判改
  - 接上一轮出题的双题对齐
  - 自由文本智能拆题
  - 对齐失败时澄清而不是乱判

这意味着这次升级已经不仅是“功能能跑”，而是：

- 工具链路能跑
- metadata 能透传到最终 done
- 记忆和 pacing 能消费逐题结果
- eval 框架也能覆盖新场景

## 面试时怎么讲

可以这样讲：

“之前我们的判题链路是单题心智模型，一次判题只返回一个总 verdict，这在真实学习场景里不够，因为学生经常会一次交两题或多题答案。于是我把 quiz_evaluator 升级成了批量判改工具，先做题答对齐，再做逐题评分。题答对齐采用的是规则优先、LLM 兜底的策略：先解析显式编号结构，再对齐最近一轮 quiz_generator 输出，如果还是不够，再让 LLM 只做结构化拆题，不直接评分。对齐成功后，工具会返回总评和逐题结果，并把每题错误单独写回错题本和知识图谱，pacing 也开始按题级结果判断。这样系统不仅能逐题批改，还把多题判改真正接进了后续的学习闭环。”
