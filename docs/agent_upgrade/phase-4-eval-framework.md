# Phase 4 - Eval Framework

## 背景问题

前面的能力升级让 Agent 看起来更像真正的学习教练，但如果没有一套稳定的评估框架，很难回答一个面试里必问的问题：

“你怎么证明它真的变好了？”

只靠人工挑几个例子演示，无法稳定评估主动推荐、replan 和节奏控制这种行为级能力。

## 为什么要这样改

这一阶段新增 Eval 框架的目的，不是为了跑单一准确率，而是为了让 Agent 的行为模式也能被系统化验证。  
尤其是下面这些能力，如果没有行为评估，几乎很难持续迭代：

- 是否触发了正确的工具
- 是否少用了不该用的工具
- 路径是不是过长或过短
- 输出有没有覆盖关键点
- 主动推荐 / replan / pacing 是否真的生效

所以这套 Eval 不只是看“答没答对”，而是看“路径和行为合不合理”。

## 核心思路

第一版评估框架采用“YAML 用例 + mock state + 真实主链路运行”的方式。

也就是说：

- 用例是结构化 YAML，便于扩展和维护
- 记忆状态、知识图谱、错题、用户画像等底层状态用 mock 注入
- 但 `agent.chat()`、planner、skill policy、tool loop、trace 这些主链路仍然真实执行

这样做的好处是：

- 能稳定复现实验场景
- 不依赖生产数据
- 测的是“系统真实行为”，而不是伪造一个单元测试结果

## 运行时链路怎么变化了

Eval 框架本身不改用户接口，而是新增一条内部测试入口：

- `tests/eval/cases/*.yaml` 定义场景
- `tests/eval/runner.py` 负责加载用例并调用 `agent.chat()`
- `tests/eval/report.py` 聚合输出结果

每条用例都可以声明：

- mock 的 knowledge_map / error_memory / profile / recent conversation
- 期望工具路径
- 期望控制模式
- 期望步数区间
- 输出必须包含的关键内容
- 是否应该触发 proactive / replan / pacing

这样一来，系统跑完之后不只是知道“过没过”，还能知道到底是：

- 工具没调对
- 工具调多了
- 路径不合理
- 输出没覆盖重点
- 或者高级行为没触发

## 关键 tradeoff

这里最主要的 tradeoff 是：第一版要优先稳定和可复现，而不是追求最接近生产。

所以目前选择的是 mock 驱动用例，而不是直接回放真实历史流量。  
这样做牺牲了一点真实世界复杂度，但换来了三件更重要的事：

- 可一键执行
- 可稳定复现
- 可明确定位失败原因

对当前阶段来说，这个取舍比“追求完全真实”更有价值，因为它更适合作为持续迭代和面试展示的基础设施。

## 效果与验证

这套框架目前重点验证四类指标：

- 工具召回率
- 工具精确率
- 路径合理性
- 输出覆盖度

同时还会附加统计：

- 主动推荐命中率
- Replan 命中率
- Pacing 命中率

这样，Phase 4 的新增能力不是“做完了但只能主观说变好了”，而是已经能用结构化用例做回归验证。

## 面试时怎么讲

可以这样讲：

“我专门补了一套 Agent 行为评估框架，不只是测答案质量，而是测行为路径。做法是用 YAML 定义场景，用 mock 的知识图谱、错题和用户画像注入状态，但真实调用 agent.chat() 跑完整主链路。这样我可以评估工具召回率、工具精确率、路径合理性和输出覆盖度，同时还能看主动推荐、replan 和 pacing 这些高级行为有没有真正触发。”
