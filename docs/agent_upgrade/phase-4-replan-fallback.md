# Phase 4 - Replan And Fallback

## 背景问题

前面的 AUTONOMOUS 模式已经能让 Agent 多步调用工具，但如果第一步结果不理想，它本质上还是“沿着原计划往下走”。  
这会导致一个很典型的问题：知识库结果不足、图谱查不到、模拟器不支持时，Agent 虽然有其他工具可用，却不会稳定地改计划。

## 为什么要这样改

真正像 Agent 的地方，不是“会调用工具”，而是“发现当前计划不够好时，会主动重排计划”。  
如果没有 Replan，系统仍然更像固定 workflow，只是多了几步工具调用。

这次改造的目标，就是让 AUTONOMOUS 模式具备真正的回退能力：

- 知识库不够时，不是一把梭结束
- 图谱查不到时，不是直接失败
- 模拟器不支持时，不是整个链路断掉
- 计算器参数不全时，不是瞎算

## 核心思路

整体策略是“先换工具或换查询，再受控降级，最后才做有限常识补充”。

这次没有把 Replan 做成独立新引擎，而是叠加在现有自主循环上：

1. 在 AUTONOMOUS prompt 中显式写入 Replan policy
2. 每次工具执行后，对结果做一次运行时评估
3. 如果发现结果不足、空结果或错误，就记录 replan signal
4. 下一轮让 Agent 带着这条 signal 重新决策

这样做的关键，是把 Replan 从“纯 prompt 希望模型自己悟到”升级成“prompt + runtime signal 双保险”。

## 运行时链路怎么变化了

改造后，自主模式下每次工具调用后都会进入一次“结果是否足够”的判断：

- `knowledge_query` 证据太少
  - 先提示尝试更宽的主题，或切到图谱 / 其他工具
- `concept_graph_query` 查不到节点
  - 先放宽主题名重试
- `protocol_state_simulator` 不支持
  - 降级为课程检索 + 文字解释
- `network_calc` 参数不足
  - 倾向于澄清参数，而不是盲算

系统会把这些信息写进 trace / metadata，比如：

- `replan_triggered`
- `replan_reason`
- `replan_count`
- `fallback_strategy`

所以后续你不仅能看到“Agent 有没有改计划”，还能看到“它为什么改”和“改成了什么策略”。

## 关键 tradeoff

这里最重要的 tradeoff 是：允许回退，但不能把回退做成无限兜底。

所以这次做了两个边界：

- Replan 只在现有步数预算内发生，不额外放宽上限
- 有限常识补充是最后一级回退，不能替代工具型结论

这意味着：

- 课程基础解释类问题，可以在证据不足时做有限补充
- 但数值计算、协议状态机、判题结果这类结论，不能靠常识补一补就算了

这能保证系统“更灵活”，但不会变成“缺工具结果时就开始自由发挥”。

## 效果与验证

这一批验证的重点是行为变化，而不是单次回答文本。

关注点包括：

- 知识库证据不足时是否真的触发了 replan
- 协议模拟器不支持时是否能自动降级
- 计算参数不合法时是否能避免胡乱给结果
- trace / metadata 里是否能清楚看到 replan 的触发和策略

这样后续做评测时，才能把“路径合理性”真正量化出来，而不是只看最终文本。

## 面试时怎么讲

可以这样讲：

“我没有把自主模式只停留在多步工具调用，而是给它加了 Plan-and-Replan 能力。做法是把回退策略同时写进 prompt 和 runtime 状态里：工具返回不足、空结果或不支持时，系统会记录 replan signal，让下一轮自主决策时重新规划路径。这样 Agent 就不是线性执行预设计划，而是能根据观察结果动态改计划，但整个过程仍然受步数和工具边界约束。”
