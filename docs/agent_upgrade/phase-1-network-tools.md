# Phase 1：Network Tools

## 背景问题

原来的工具已经能完成知识检索、复习总结、出题和判题，但这些能力更像“学习流程工具”，还缺少真正体现计算机网络专业性的分析型工具。

结果就是：

- 计算题容易退回到大模型“脑算”
- 协议过程题只能靠文字解释
- 知识依赖和薄弱点分析缺少结构化支撑

## 为什么要这样改

如果想让 Agent 真正体现“自主选工具”的价值，就必须让工具池里有一些**只有工具才能稳定完成**的动作。

在计算机网络这个领域，最典型的三类就是：

- 精确数值计算
- 知识结构查询
- 协议状态模拟

所以这次新增了三类专业工具，而不是继续堆更多泛用文本工具。

## 核心思路

新增的三个工具分别承担不同角色：

- `network_calc`
  - 负责子网划分、CRC、滑动窗口、吞吐/时延、香农/奈奎斯特等精确计算
- `concept_graph_query`
  - 负责查询知识点的前置依赖、子主题、易混淆概念和推荐复习顺序
- `protocol_state_simulator`
  - 负责 TCP 握手/挥手、拥塞控制、故障注入和路由更新等可结构化模拟的过程

它们的设计原则是统一的：

- 能确定算出来的，不交给大模型脑补
- 能结构化表示的，不只返回松散文本
- 大模型负责决策、组织和表达，工具负责精确分析

## 运行时链路怎么变化了

加入新工具后，复杂学习任务不再只能在“检索、总结、出题、判题”几个高层工具之间跳转。  
Agent 在自主模式里可以多一种层次：

- 先查知识图谱掌握度
- 再检索课件证据
- 必要时调用状态模拟器
- 最后再组织成用户可读解释

这让系统从“只会查资料”变成“会分析、会判断、会演示”。

## 关键 tradeoff

这里最大的 tradeoff 是：**只加课程专用工具，不引入通用执行器**。

这样做的好处：

- 工具语义清晰
- 面试时容易讲出专业性
- 风险低，不会演变成不可控的泛化代理

代价是：

- 可覆盖的任务边界更窄
- 需要手工设计部分工具接口和结果格式

但这和项目定位一致，因为目标是“计算机网络学习 Agent”，不是通用多领域代理。

## 效果与验证

这三个工具让系统具备了以前没有的动作空间：

- 计算题可以主动调用 `network_calc`
- 结构化复习可以先看 `concept_graph_query`
- 协议推演题可以直接交给 `protocol_state_simulator`

这部分的验证重点不是大模型评分，而是：

- 工具本身输出是否稳定
- 是否能在 Agent trace 里看到被自主选中
- 是否能支撑后续的学习策略层

## 面试时怎么讲

可以直接这样讲：

> 我增加了三类计算机网络领域专用工具，目的不是单纯扩工具数量，而是把大模型不擅长、但教学中又非常关键的能力抽出来交给工具做，比如子网划分和 CRC 这种精确计算、知识点前置依赖查询、以及 TCP 状态机和拥塞控制过程模拟。这样 Agent 在复杂任务里就不是只会查知识库，而是真正可以根据场景选择“查图谱、算一遍、或者模拟一遍”。
