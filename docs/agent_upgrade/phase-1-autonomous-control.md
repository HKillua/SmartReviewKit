# Phase 1：Autonomous Control

## 背景问题

原来的系统已经有 Agent 和工具调用，但整体还是偏强工作流：  
大多数请求会被识别成某个固定意图，再尽量绑定到某个固定工具上。这样做的好处是稳，但问题是 Agent 的“自主决策感”不够强，复杂学习任务很难体现“先观察，再决定下一步”的过程。

## 为什么要这样改

学习场景里并不是所有问题都适合固定流程。

- 简单问答适合快路径
- 复习、错题复盘、章节深入讲解这类任务更像“多步推理 + 多次观察”

如果所有任务都走固定工具绑定，会让系统更像 workflow；  
如果所有任务都放开自由多步，又会失控。  
所以需要新增一个**受控自主层**，只在复杂学习任务里开放多步自主决策。

## 核心思路

核心做法是把控制模式从三档扩成四档：

- `PASS_THROUGH`
- `ADVISORY`
- `FORCE_TOOL`
- `AUTONOMOUS`

`AUTONOMOUS` 不是放飞，而是被强约束的自主模式：

- 只有命中允许自主的 Skill 或显式复合学习任务时才开放
- 单次最多 5 步
- 同一工具最多调用 2 次
- `document_ingest` 永不进入自主模式

也就是说，系统不是“全面改成 Agent”，而是“在复杂任务里切入受控自主”。

## 运行时链路怎么变化了

改造前：

用户问题 → Planner 识别意图 → 固定工具 / 固定工作流 → 输出结果

改造后：

用户问题 → Skill 命中与否 → Planner 识别任务 → 如果是简单任务继续快路径  
如果是复杂学习任务，则进入：

Thought → Action → Observe → 再决定下一步

并且整个过程中，SkillPolicy 会持续约束：

- 哪些工具能用
- 最多走几步
- 什么时候该停

## 关键 tradeoff

这次最关键的取舍是：**只把复杂任务做成自主模式，简单任务不牺牲效率**。

这样做的优点：

- 复杂任务能体现 Agent 感
- 简单问答不会因为追求自主性而显著变慢
- 工具空间增大后仍然可控

代价是：

- 这不是完全通用 Agent
- 自主路径是“按场景授权”的，不是默认全开放

但这个取舍是合理的，因为项目定位本来就不是通用代理，而是“计算机网络学习 Agent”。

## 效果与验证

第一阶段最重要的效果不是指标，而是**运行方式变化**：

- 系统具备了多轮 Tool Loop 的能力
- Trace 中可以明确看到 `autonomous_iteration`
- 最终回答里会带 `tool_path`、`autonomous_step_count`、`autonomous_stop_reason`

这意味着项目叙事已经从“工具化问答系统”升级成了“受控自主的学习 Agent”。

## 面试时怎么讲

可以直接这样讲：

> 我没有把系统直接改成完全放飞的通用代理，而是增加了一个 `AUTONOMOUS` 模式，只在复杂学习任务中开放多步自主决策。简单任务继续走高效快路径，复杂任务才进入 Thought-Action-Observe 循环。这样既保留了 Agent 的自主感，又通过工具白名单、步数上限和禁止工具规则把系统约束在教学场景里。
