# Phase 6 - Simplified Planner Refactor

## 1. 背景问题

前面的 Agent 演进里，Skill 已经成为学习场景的策略层，但 Planner 仍然保留了较重的前置决策逻辑。  
问题在于，一旦整句请求命中 Skill，Planner 很容易过早被短路，导致用户显式表达的“先 A 再 B”结构被吞掉。

这会带来两个直接问题：

- 复合学习任务的顺序信息丢失，系统更像在“猜一个总场景”，而不是尊重用户原始意图
- Skill 和 Planner 的职责开始重叠，系统越来越难解释，也越来越难稳定扩展

因此，这一阶段的目标不是让 Planner 更聪明，而是让它更简单、更保守、更可解释。

## 2. 为什么要这样改

这次重构没有继续往复杂的 `skill_scope` 架构方向推进，而是故意收敛成一个低风险 v1。

原因很现实：

- 复合任务结构识别如果做得太激进，很容易误拆
- Skill 覆盖范围如果做得太复杂，很容易让 runtime 变得难以调试
- 这个项目已经有较强的工具和 Skill 能力，Planner 不应该再变成第二套“大脑”

所以这次的取舍是：

- 只支持显式顺序复合任务
- 只复用现有单任务识别能力，不新造分类器
- Skill 只作为一个连续区间覆盖，不引入多段、多 Skill 联动
- 模糊请求默认先保守推进，而不是上来就反问用户

## 3. 核心思路

这一版 Planner 的思路可以概括成三句话：

1. 先保住用户表达结构  
   只有当消息里出现明显的顺序词，如“先 / 再 / 然后 / 最后”时，才进入复合任务路径。

2. 复用现有能力做逐段识别  
   每个 segment 都继续走现有的 `rule -> embedding -> default` 链，不额外引入新的模型或复杂路由器。

3. Skill 只做后缀区间覆盖  
   找到第一个命中 Skill 的 segment，再看后续 subtasks 的工具是否仍在 `allowed_tools` 内；如果是，就继续覆盖；否则立即停止。

这意味着系统从“Skill 一命中就吞掉整句”变成了“先拆出 agenda，再决定 Skill 从哪一段开始生效”。

## 4. 运行时链路怎么变化了

### 旧链路

- 用户消息先命中 Skill
- Planner 早期短路成一个总决策
- 后面再用 `skill_policy` 做一次修正

结果是：

- 简单任务能跑
- 但复合任务里显式顺序很容易丢

### 新链路

- 先检查是否有显式顺序结构
- 如果有，就先切成多个顺序 segments
- 每个 segment 独立识别任务类型和初始工具
- 再对这些 segment 计算一个连续 Skill 区间
- runtime 执行时：
  - 区间前缀按普通 composite 执行
  - Skill 区间内按 `skill_policy` 约束执行
  - 区间结束后回到普通 composite 执行

如果整句没有显式顺序结构，系统完全回退到原来的单任务 Planner 路径。

## 5. 关键 tradeoff

这次最重要的 tradeoff 是“主动降维”。

我们没有在 v1 里做：

- 并列多主题拆分
- 隐式复合任务理解
- 多 Skill 共存
- 基于 topic / output contract 的复杂连续性打分

这样做的代价是：

- 一些复杂自然语言复合任务还不会被拆得很细
- Planner 仍然偏保守

但好处非常明确：

- 技术风险显著下降
- 复合任务只在最容易识别的场景里触发
- Skill 和 Planner 的职责边界开始清晰
- 后续如果要继续增强，可以在这条链路上逐步加能力，而不是先上一个很重的框架

## 6. 效果与验证

这一版重点验证的是行为正确性，而不是能力上限。

核心覆盖包括：

- 没有显式顺序词时，Planner 维持原有单任务行为，不回归
- 有显式顺序词时，能够稳定切出顺序 segments
- 每个 segment 复用现有 `rule / embedding / default` 链
- Skill 不再全局污染前缀 segment，只在连续区间内生效
- 后续工具一旦越出 `allowed_tools`，Skill 覆盖立即停止
- 低风险模糊请求继续保守推进，高风险缺参场景仍然走澄清

所以它解决的是“结构被吞掉”的核心问题，而不是一口气做最强的任务理解。

## 7. 面试时怎么讲

这次我没有把 Planner 往更复杂的方向继续堆，而是反过来把它简化了。因为前面的问题不是 Planner 不够聪明，而是 Skill 一命中就太容易把显式复合结构吞掉。  
所以我做的调整是：只支持显式顺序复合任务，先按“先 / 再 / 然后 / 最后”切成多个 segments，然后对每个 segment 继续复用原来的 `rule -> embedding -> default` 意图识别链，再让 Skill 只作为一个连续覆盖区间生效。这样前缀子任务不会被全局 Skill 污染，后缀仍然可以享受 Skill 带来的工具约束和自主能力。  
这次重构的核心不是追求更强的理解，而是把 Planner 收敛成一个更保守、更可解释、更容易扩展的前置结构整理器。
