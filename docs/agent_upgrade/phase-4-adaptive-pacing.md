# Phase 4 - Adaptive Pacing

## 背景问题

即使前面已经有了工具、自主模式和 SkillPolicy，如果系统对所有学生都用同一种讲解节奏，它依然不像真正的教学 Agent。  
现实中的老师会根据学生最近做题表现和追问方式，动态决定是加快推进还是放慢解释。

## 为什么要这样改

学习过程里的“节奏控制”其实很关键：

- 学得快的学生不想一直听基础铺垫
- 学得慢或连续做错的学生，如果还被快速推进，会越学越乱

如果 Agent 能根据近期表现动态调节讲解方式，它就不只是“回答对”，而是“教得更合适”。

## 核心思路

这次没有引入新的评分模型，而是用两类很便宜但有教学意义的信号来估计节奏：

1. 最近几次做题表现
   - 连续做对，说明可以推进得更快
   - 连续做错，说明需要放慢
2. 最近对话风格
   - 短而困惑的回复，说明可能跟不上
   - 连续追问“为什么 / 更深入”，说明可以讲深一些

系统把这些信号压成 3 档节奏：

- `accelerate`
- `normal`
- `decelerate`

然后把结果重新注入 AUTONOMOUS prompt，让 Agent 在当前回合自然调整讲解风格。

## 运行时链路怎么变化了

改造后，自主模式每一轮都会先估计当前学习节奏：

- 如果最近判题连续正确，Agent 会更少重复基础内容，更快推进到高阶点
- 如果最近判题连续错误，或者用户回复明显很短、很困惑，Agent 会放慢速度，多给例子，必要时先补前置知识

同时，这个节奏结果不会只存在内存里。  
在消息结束后，系统会把本轮推断出的 `learning_pace` 写回 student profile，这样它就不只影响当前一轮，也会对后续会话产生连续作用。

trace / metadata 里会新增：

- `pacing_level`
- `pacing_reason`

所以后续可以直接看到系统为什么把某个用户判成“该加速”或“该放慢”。

## 关键 tradeoff

这部分最大的 tradeoff 是：节奏判断必须足够便宜，但不能完全拍脑袋。

因此这次用的是“最近 quiz 结果 + 对话信号”的组合，而不是引入额外模型。  
这样做的优点是成本低、可解释、容易稳定；缺点是它不可能像完整学习分析系统那样精细。

但对第一版来说，这个取舍是合理的，因为它已经能把系统从“统一节奏”升级成“有方向性的动态节奏”。

## 效果与验证

这批验证关注的是：

- 连续做对时，是否会进入 `accelerate`
- 连续做错时，是否会进入 `decelerate`
- 没有做题记录时，短消息和深追问是否也能触发节奏变化
- `learning_pace` 是否真的被写回用户画像

这意味着后续不只是能看回答文本，还能看系统对学生当前学习状态的判断是否合理。

## 面试时怎么讲

可以这样讲：

“我给 Agent 增加了学习节奏自适应能力。它会根据最近几次做题结果和对话风格，把当前节奏判成 accelerate、normal 或 decelerate，然后把这个节奏信号注入自主模式的 prompt，控制回答是更快推进还是放慢解释。同时我把这个节奏结果写回 student profile，让它成为跨会话的长期学习特征，而不是一次性的临时变量。”
