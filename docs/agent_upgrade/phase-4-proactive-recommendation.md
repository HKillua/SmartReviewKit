# Phase 4 - Proactive Recommendation

## 背景问题

前几个阶段已经把系统做成了“用户问什么，Agent 就处理什么”的受控自主学习 Agent，但它仍然是被动的。  
真实的学习教练不会一直等学生发出完整指令，而是会在学生只说“你好”“在吗”这类低信息量开场时，主动观察学习状态并提出最有价值的下一步建议。

## 为什么要这样改

如果系统只能在用户明确下达任务后才行动，它更像一个带工具的问答机，而不是学习教练。  
主动推荐的价值在于：

- 把长期记忆真正转化成行动建议，而不是只在回答里偶尔被动引用
- 提升低交互门槛场景下的体验，让学生一上线就能获得“我下一步该学什么”的指引
- 让面试叙事从“有记忆”升级为“会利用记忆主动干预学习流程”

## 核心思路

这次没有单独新增一套并行的主动检查入口，而是把主动推荐并入现有的主动复习钩子链路。  
整体思路是：

1. 先判断用户消息是不是低信息量
2. 如果不是，就完全不干扰正常 Planner
3. 如果是，就从知识图谱衰减、高频弱项、最近活跃时间三类信号里挑高价值信息
4. 把这些信号整理成一段 `[Proactive Insight]` 注入系统上下文
5. 由 Agent 决定如何自然地把推荐说出来

这种设计的好处是，主动推荐仍然是“上下文增强”，不是“强行改写用户输入”，所以不会破坏原来的任务识别链路。

## 运行时链路怎么变化了

改造后，用户消息进入系统时会先经过主动复习钩子：

- 对“帮我讲 TCP”这类专业问题，钩子直接放行
- 对“你好”“hi”“在吗”这类低信息量开场，钩子会额外读取：
  - 衰减后的知识图谱弱项
  - 高频错题对应的弱概念
  - 最近活跃时间

如果这些信号足够明确，系统会在不改写用户输入的前提下，给 Agent 一段推荐上下文，让它自然地做出像“你已经几天没复习拥塞控制了，要不要快速过一遍”的响应。

同时，trace 和最终 metadata 里会明确记录这次有没有触发主动推荐、触发原因是什么、依据了哪些信号。

## 关键 tradeoff

这块最大的取舍是：主动推荐必须增强体验，但不能打断正常任务。

因此这次做了两个约束：

- 只对低信息量开场触发，不对明确学习问题触发
- 只挑高价值弱项，不一次性推太多知识点，避免把开场做成“学习状态报表”

这意味着系统不会在每次对话里都显得很“主动”，但换来的好处是更稳、更自然，也更容易解释为什么它会在某些时刻主动提醒、在另一些时刻完全不打扰。

## 效果与验证

这次验证的重点不是“回复变长了”，而是：

- 低信息量消息可以触发主动推荐
- 专业问题不会误触发
- 知识衰减逻辑会在推荐前先刷新数据
- trace / metadata 能保留 `proactive_triggered`、`proactive_reason` 和信号来源

这样后续无论是做产品分析还是做评估，都能明确知道主动推荐到底有没有被触发、为什么触发。

## 面试时怎么讲

可以直接讲成一句完整的话：

“我把主动推荐并入了原来的主动复习钩子链路，没有另起一套平行入口。只有当用户发的是‘你好’这类低信息量消息时，系统才会读取知识图谱衰减、高频错题和最近活跃时间，生成一段主动学习提示注入上下文。这样 Agent 就能像学习教练一样在合适的时候主动给出复习建议，但不会干扰正常的专业问答。”
